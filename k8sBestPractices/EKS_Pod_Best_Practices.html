Resource Management
CPU & Memory requests and limits — the foundation of a stable cluster
P1 Critical
⚠️
Missing resource limits is the #1 cause of node crashes and OOMKill in production EKS clusters. Every container must have both requests and limits defined.

What are Requests?

The minimum guaranteed resources the pod needs. Kubernetes uses this to decide which node to schedule the pod on.

What are Limits?

The maximum resources a pod can consume. If exceeded, CPU is throttled. If memory is exceeded, pod is OOMKilled.

Correct Configuration

resources:
  requests:
    memory: "256Mi"   # guaranteed allocation
    cpu: "250m"       # 250 millicores = 0.25 CPU
  limits:
    memory: "512Mi"   # max — OOMKill if exceeded
    cpu: "500m"       # max — throttled if exceeded

Health Checks (Probes)
Liveness · Readiness · Startup — tell Kubernetes when your pod is healthy
P1 Critical

Three Probes — Three Different Jobs

ProbeQuestion it answersFailure actionSpring Boot endpoint
StartupHas the app finished initialising?Keep waiting (don't restart yet)/actuator/health
ReadinessIs the app ready to receive traffic?Remove from load balancer/actuator/health/readiness
LivenessIs the app still alive?Restart the pod/actuator/health/liveness

Configuration

livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 30   # wait 30s before first check
  periodSeconds: 10          # check every 10s
  failureThreshold: 3        # restart after 3 consecutive failures

readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 20
  periodSeconds: 5
  failureThreshold: 3        # remove from LB after 3 failures

startupProbe:
  httpGet:
    path: /actuator/health
    port: 8080
  failureThreshold: 30       # allow up to 300s (30 × 10s) to start
  periodSeconds: 10
🚨
Never skip readiness probe. Without it, Kubernetes sends traffic to your pod the moment it starts — before Spring Boot has loaded beans, connected to DB, or warmed up caches. Users see 503 errors for the first 20–30 seconds of every deployment.

Pod Disruption Budget (PDB)
Guarantee minimum availability during node drains, upgrades, and rolling deploys
P2 Important

Without PDB — Risk of Downtime

With PDB — Zero Downtime

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: myapp-pdb
  namespace: production
spec:
  minAvailable: 2        # always keep at least 2 pods running
  selector:
    matchLabels:
      app: my-spring-boot-app

Horizontal Pod Autoscaler (HPA)
Automatically scale pods up and down based on CPU and memory usage
P2 Important

How HPA Works

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
  namespace: production
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: my-spring-boot-app
  minReplicas: 2          # never go below 2 (HA guarantee)
  maxReplicas: 20         # cap to control AWS costs
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70    # scale up when avg CPU > 70%
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80    # scale up when avg memory > 80%

Security Context — Never Run as Root
Contain the blast radius of any pod compromise
P1 Critical
❌ Running as Root — Danger
✅ Non-Root — Contained
securityContext:
  runAsNonRoot: true              # pod fails to start if image runs as root
  runAsUser: 1000                 # run as specific non-root user
  runAsGroup: 1000
  readOnlyRootFilesystem: true    # filesystem is read-only — attacker can't write
  allowPrivilegeEscalation: false # cannot gain more privileges than parent
  capabilities:
    drop:
      - ALL                       # drop all Linux capabilities

Secret Management — Never Hardcode
Use IRSA + Secrets Manager for zero credential leakage
P1 Critical
❌ Wrong — Hardcoded
env:
  - name: DB_PASSWORD
    value: "MyP@ssw0rd123"
# Stored in Git forever
# Anyone with repo access
# can see this password
✅ Correct — Secrets Manager
env:
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: aurora-secret
        key: password
# Fetched from Secrets Manager
# Rotates automatically
# Never in Git

Network Policy — Zero Trust
Default deny all traffic, then explicitly allow only what is needed
P3 Next Sprint

Zero Trust Network Model

# Step 1 — Default deny ALL ingress and egress
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}    # applies to all pods in namespace
  policyTypes: [Ingress, Egress]

---
# Step 2 — Allow only what the microservice needs
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: myapp-allow
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: my-spring-boot-app
  ingress:
    - from:
        - podSelector:
            matchLabels:
              role: api-gateway    # only gateway can call me
      ports:
        - port: 8080
  egress:
    - to:
        - podSelector:
            matchLabels:
              app: aurora-endpoint # I can only call Aurora
      ports:
        - port: 5432
    - ports:
        - port: 53                 # always allow DNS

Pod Anti-Affinity — Spread Across Nodes
Never put all eggs in one basket — spread pods across nodes and AZs
P2 Important
❌ Without Anti-Affinity
✅ With Anti-Affinity
affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
      - labelSelector:
          matchExpressions:
            - key: app
              operator: In
              values: [my-spring-boot-app]
        topologyKey: kubernetes.io/hostname    # spread across nodes
  # Also spread across Availability Zones
  podAntiAffinity:
    preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        podAffinityTerm:
          labelSelector:
            matchLabels:
              app: my-spring-boot-app
          topologyKey: topology.kubernetes.io/zone  # spread across AZs

Graceful Shutdown
Drain connections before pod dies — zero dropped requests on every deployment
P2 Important

Shutdown Sequence

# Pod spec
terminationGracePeriodSeconds: 60
lifecycle:
  preStop:
    exec:
      command: ["sh", "-c", "sleep 10"]   # wait for LB to deregister

# application.yml
server:
  shutdown: graceful
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s       # max time to drain requests

Deployment Strategy
Zero downtime rolling updates with maxUnavailable: 0
P2 Important 
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1        # create 1 extra pod before removing old ones
    maxUnavailable: 0  # never take a pod down before replacement is ready
                       # this combination = zero downtime guaranteed

Complete Production-Ready Pod Template
Copy-paste ready — all best practices combined into one manifest
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-spring-boot-app
  namespace: production
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0
  selector:
    matchLabels:
      app: my-spring-boot-app
  template:
    metadata:
      labels:
        app: my-spring-boot-app
      annotations:
        prometheus.io/scrape: "true"
        prometheus.io/port: "8080"
        prometheus.io/path: "/actuator/prometheus"
    spec:
      serviceAccountName: myapp-sa           # IRSA role attached here
      terminationGracePeriodSeconds: 60
      affinity:
        podAntiAffinity:
          requiredDuringSchedulingIgnoredDuringExecution:
            - labelSelector:
                matchExpressions:
                  - key: app
                    operator: In
                    values: [my-spring-boot-app]
              topologyKey: kubernetes.io/hostname
      containers:
        - name: my-spring-boot-app
          image: 123456789.dkr.ecr.ap-south-1.amazonaws.com/myapp:1.0.5
          ports:
            - containerPort: 8080
          resources:
            requests:
              memory: "256Mi"
              cpu: "250m"
            limits:
              memory: "512Mi"
              cpu: "500m"
          securityContext:
            runAsNonRoot: true
            runAsUser: 1000
            readOnlyRootFilesystem: true
            allowPrivilegeEscalation: false
            capabilities:
              drop: [ALL]
          env:
            - name: DB_HOST
              valueFrom:
                secretKeyRef:
                  name: aurora-secret
                  key: host
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: aurora-secret
                  key: password
          livenessProbe:
            httpGet:
              path: /actuator/health/liveness
              port: 8080
            initialDelaySeconds: 30
            periodSeconds: 10
            failureThreshold: 3
          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: 8080
            initialDelaySeconds: 20
            periodSeconds: 5
            failureThreshold: 3
          startupProbe:
            httpGet:
              path: /actuator/health
              port: 8080
            failureThreshold: 30
            periodSeconds: 10
          lifecycle:
            preStop:
              exec:
                command: ["sh", "-c", "sleep 10"]
